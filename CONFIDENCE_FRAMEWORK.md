# Confidence Framework Guide

How to add new EOB analysis rules while maintaining algorithm trustworthiness.

## Overview

Every `SavingsOpportunity` generated by the analyzer includes confidence metadata that helps users understand how certain we are about each finding. This framework ensures new rules follow the same guardrails.

**Key Principle:** Better to be conservative than to mislead. Users trust us with decisions about expensive appeals.

---

## System Architecture

### SavingsOpportunity Fields

Every rule must populate these fields:

| Field | Type | Purpose | Example |
|-------|------|---------|---------|
| `opportunity_type` | str | Category of savings | `"duplicate_charge"` |
| `estimated_savings` | float | Dollar amount at stake | `450.00` |
| `plan_claim_numbers` | list[str] | Which claims are affected | `["CLM-001", "CLM-002"]` |
| `confidence_score` | float | 0.0–1.0, quantifies certainty | `0.75` |
| `confidence_level` | str | Human-readable tier | `"High"` / `"Medium"` / `"Low"` |
| `flag_reason` | str | Plain English why we flagged this | `"Identical charge appears twice with 47-min gap"` |
| `verification_steps` | list[str] | Checklist user should verify before acting | `["Call provider to confirm duplicate", "Check claim adjudication dates"]` |
| `could_be_correct_if` | list[str] | Scenarios where finding might be a false positive | `["Charges are for different service dates", "Deductible applied twice intentionally"]` |
| `missing_data_points` | list[str] | Data gaps lowering confidence | `["No member ID on EOB", "Service date unclear"]` |
| `recommended_action` | str | Next step user should take | `"Contact provider within 30 days of EOB date"` |

---

## Confidence Scoring Guidelines

### Scoring Scale

- **0.85–1.0:** High confidence. Rule fires on clear-cut patterns (exact duplicates, OON balance billing over allowed amount, denied service w/ appeal rights).
- **0.65–0.84:** Medium confidence. Rule has edge cases or requires user judgment (potential coding errors, disputed charges with partial support).
- **0.40–0.64:** Low confidence. Rule is speculative or highly context-dependent (unusual patterns, possible plan misunderstanding).
- **Below 0.40:** Do not flag. Too risky; mention only in analytics, not in results UI.

### Rule Examples

**Duplicate Charge (Confidence: 0.86)**
```python
if same_claim_line_appears_twice_within(gap=timedelta(hours=2)):
    confidence = 0.86
    flag_reason = f"Identical charge ({amount}) appears twice with {gap} gap"
    verification_steps = [
        "Call provider EOB department; ask if duplicate is processing error",
        "Cross-check against your internal payment records"
    ]
```

**OON Balance Billing (Confidence: 0.70)**
```python
if charge_network_status == "OON" and balance_bill > member_expected:
    confidence = 0.70  # Out-of-network but requires user to confirm balance bill rules
    flag_reason = "Out-of-network charge appears to exceed balance bill limits"
    could_be_correct_if = [
        "Your plan allows balance billing up to this amount",
        "Charge is bundled with in-network services"
    ]
```

**Appeal-Worthy Denial (Confidence: 0.68)**
```python
if denial_code in APPEALABLE_CODES and medical_necessity_support_exists:
    confidence = 0.68  # Medical necessity is subjective
    flag_reason = f"Service denied but may be appealable (reason: {denial_code})"
    verification_steps = [
        "Gather provider's clinical notes justifying medical necessity",
        "Check your plan's appeal process and timeline"
    ]
```

## Data-Gap Guardrails

If critical plan information is missing from the EOB, we downgrade confidence to prevent overconfident guidance:

```python
def apply_data_confidence_guard(opp: SavingsOpportunity, eob_data: dict) -> SavingsOpportunity:
    """
    Lower confidence if we're missing key data needed to judge savings opportunities.
    """
    data_gaps = []
    
    if not eob_data.get("member_deductible_status"):
        data_gaps.append("Deductible status not found")
    if not eob_data.get("member_oop_met"):
        data_gaps.append("Out-of-pocket status not found")
    if not eob_data.get("plan_name") or "unknown" in eob_data.get("plan_name", "").lower():
        data_gaps.append("Plan type unclear")
    if not eob_data.get("member_id"):
        data_gaps.append("Member ID not found")
    
    opp.missing_data_points = data_gaps
    
    # Confidence guardrail: penalize if too many gaps
    if len(data_gaps) >= 4:
        opp.confidence_score = min(0.55, opp.confidence_score)
        if opp.confidence_score < 0.55:
            opp.confidence_level = "Low"
    elif len(data_gaps) >= 3:
        opp.confidence_score = min(0.69, opp.confidence_score)
        if opp.confidence_score > 0.69:
            opp.confidence_level = "Medium"
    
    return opp
```

**When to trigger guardrails:**
- Missing deductible/OOP accumulator status → Can't judge if duplicate is truly duplicate
- Missing service dates → Can't distinguish same-day from different-day charges
- Missing plan type → Can't apply plan-specific rules (copay vs. deductible vs. coinsurance)
- Missing member ID → Suggests low-quality EOB; less trustworthy entire result set

## Adding a New Rule: Validation Checklist

Before committing a new rule to `eob_analyzer.py`:

- [ ] **Accuracy:** Rule produces **zero false positives** on your family EOBs (10+ test cases)
- [ ] **Confidence Score:** Score reflects actual finding certainty (not optimistic wish-thinking)
- [ ] **Flag Reason:** Written in plain English, a non-tech user understands *why* it was flagged
- [ ] **Verification Steps:** At least 2 actionable steps user can take to validate before acting
- [ ] **Could-Be-Correct-If:** At least 1 scenario where flag might be false alarm
- [ ] **Recommended Action:** Clear next step (e.g., "Call provider within 30 days", "Gather medical notes")
- [ ] **Data Gaps:** Rule identifies which missing EOB fields lower its confidence
- [ ] **Unit Test:** `test_eob_analyzer.py` includes case where rule fires correctly + case where rule correctly *doesn't* fire
- [ ] **Comment:** Code comment explains why this confidence score was chosen
- [ ] **No Snake Oil:** Ask yourself: "If this advice leads to a failed appeal, can I defend it?" If not, lower confidence or don't ship.

## Code Structure Example

In `app/services/eob_analyzer.py`:

```python
def _identify_savings_opportunities(eob_data: dict) -> list[SavingsOpportunity]:
    opportunities = []
    
    # RULE: Duplicate Charges
    duplicates = find_duplicate_charges(eob_data)
    for dup_group in duplicates:
        opp = SavingsOpportunity(
            opportunity_type="duplicate_charge",
            estimated_savings=dup_group["amount"],
            plan_claim_numbers=dup_group["claim_ids"],
            confidence_score=0.86,
            confidence_level="High",
            flag_reason=f"Identical charge (${dup_group['amount']}) appears {len(dup_group['claims'])}x within 2 hours",
            verification_steps=[
                "Call provider -> ask if duplicate is processing error",
                "Check your bank/insurance portal for payment status on each claim",
                "If unsure, request provider re-adjudicate one claim"
            ],
            could_be_correct_if=[
                "Charges are for two different service dates (e.g., office visit + lab same day)",
                "One charge is copay, other is deductible"
            ],
            missing_data_points=[],  # Will be populated by apply_data_confidence_guard()
            recommended_action="Call provider within 30 days of EOB date"
        )
        opportunities.append(opp)
    
    # Apply data-gap guardrails
    opportunities = [apply_data_confidence_guard(opp, eob_data) for opp in opportunities]
    
    return opportunities
```

## Testing & Iteration

### Unit Test Template

```python
def test_duplicate_charge_detection():
    """Rule should fire on exact duplicates within 2 hours."""
    eob_with_dup = {
        "claims": [
            {"_id": "C1", "amount": 450, "provider": "Acme Hospital", "date": "2024-01-15 10:00"},
            {"_id": "C2", "amount": 450, "provider": "Acme Hospital", "date": "2024-01-15 10:47"}
        ]
    }
    opps = _identify_savings_opportunities(eob_with_dup)
    
    assert len(opps) == 1
    assert opps[0].opportunity_type == "duplicate_charge"
    assert opps[0].confidence_score == 0.86
    assert "Call provider" in opps[0].verification_steps[0]

def test_different_service_dates_not_flagged():
    """Rule should NOT fire when service dates differ."""
    eob_no_dup = {
        "claims": [
            {"_id": "C1", "amount": 450, "provider": "Acme Hospital", "date": "2024-01-15"},
            {"_id": "C2", "amount": 450, "provider": "Acme Hospital", "date": "2024-01-16"}
        ]
    }
    opps = _identify_savings_opportunities(eob_no_dup)
    
    assert len([o for o in opps if o.opportunity_type == "duplicate_charge"]) == 0
```

### Real-World Validation

Once you deploy a new rule:
1. Run it on 50+ de-identified family EOBs
2. For each finding, manually verify: "Would a real person save money acting on this?"
3. Track appeal outcomes: did users who acted get refunded? Did non-users miss refunds?
4. Adjust confidence thresholds based on real results

## Anti-Patterns (Do Not Do)

❌ **High confidence without verification steps**
```python
# DON'T DO THIS
confidence_score = 0.95
verification_steps = []  # <-- Users have no way to validate
```

❌ **Vague flag reasons**
```python
# DON'T DO THIS
flag_reason = "Issue detected"  # <-- User doesn't know *what* issue
```

❌ **Missing data gaps**
```python
# DON'T DO THIS
missing_data_points = []  # <-- Even if empty, still run apply_data_confidence_guard()
```

❌ **Scoring on gut instinct**
```python
# DON'T DO THIS
if weird_pattern_detected:
    confidence_score = 0.80  # <-- Why 0.80? Just feels right?
    # INSTEAD: Score should reflect your actual error rate on test cases
```

## Questions?

Before shipping a rule:
1. **Can I defend this in an appeal hearing?** (If not, lower confidence or kill the rule)
2. **Do I have at least 10 real test cases?** (If not, test more)
3. **Would I flag this on my own EOB?** (If not, maybe it's too speculative)
4. **What's the worst-case outcome if I'm wrong?** (Cost of false positive should be low vs. savings at stake)

**Our mission:** Help users find *real* money, not manufacture outrage over false positives.
